# MES 프로젝트: 요청 분류 및 처리 최종 가이드

고객요청, 이슈, 협의요청을 정확히 분류하고 효율적으로 처리하는 방법

---

## 📌 3가지 분류 정의

### 1️⃣ 고객요청 (Customer Request)

#### 정의
- 고객이 원하는 **새로운 기능/프로세스 변경**
- 계약서에 명시되거나 협의 과정에서 나온 요구사항

#### 특징
- 설계 변경 필요 (기능 레벨)
- 비용/일정 영향 큼
- 계약 범위 내/외 판단 필요
- 변경관리 프로세스 필수

#### 상태 흐름
```
REVIEWING (검토)
    ↓
APPROVED (승인) / REJECTED (거절)
    ↓
IN_DEVELOPMENT (개발)
    ↓
APPLIED (적용) / HOLD (보류)
```

#### 처리자
PM + 고객 (변경관리 위원회)

#### 처리 기간
1~4주 (협의 + 승인)

#### 예시
- "IVI공정에서 수량 검수 자동화 기능 추가"
- "보고서 형식을 PDF로 변경해달라"
- "기계 5대 추가로 MES 연결"

---

### 2️⃣ 이슈 (Issue)

#### 정의
- 설계된 기능이 **제대로 작동하지 않는 버그/결함**
- 명확한 오류 발생, 재현 가능

#### 특징
- 코드 수정으로 해결 가능
- 심각도 기반 우선순위
- 계약 범위 내 (품질 문제)
- 즉시 처리 필요

#### 상태 흐름
```
OPEN (발견)
    ↓
IN_PROGRESS (수정 중)
    ↓
RESOLVED (해결) / WONT_FIX (수정 안함)
    ↓
CLOSED (완료)
```

#### 처리자
개발팀 (기술 담당자)

#### 응답 기준
| 심각도 | 응답 시간 |
|--------|----------|
| CRITICAL | 즉시 (1시간) |
| HIGH | 24시간 |
| MEDIUM | 1주 |
| LOW | 차기 버전 |

#### 예시
- "로그인 화면에서 특수문자 비밀번호 오류" (재현율 100%)
- "로그 시간이 UTC로 저장됨"
- "바코드 스캔이 간헐적으로 실패"

---

### 3️⃣ 협의요청 (Discussion Item)

#### 정의
- **버그도 아니고 새 기능도 아닌데, 어떻게 할지 결정이 필요한 사항**
- 고객과 의사결정을 거쳐 최종 처리 방향 결정

#### 특징
- 어느 단계에서든 발생 (분석/설계/구현/테스트/운영)
- 여러 선택지 존재
- 기술적 제약/한계 포함 가능
- 최종적으로 고객요청 또는 BLOCKED로 변환

#### 상태 흐름
```
DISCUSSING (협의 중)
    ↓
(고객 의사결정)
    ↓
→ 고객요청 변환 (REQ-XXXX)
→ 업무협조 분해 (COOP-XXXX)
→ BLOCKED (지연/미적용)
→ 종료 (결정완료)
```

#### 처리자
기술팀 + PM + 고객

#### 처리 기간
2~7일 (분석 + 협의)

#### 예시
- "데이터 시간 기준이 다른데 어떻게 통합할까?"
- "구형 설비라 실시간 수집 불가능한데 어떻게?"
- "화면 레이아웃 변경 요청이 들어왔는데 가능한가?"
- "데이터 오버플로우 위험이 3년 후에 발생하는데?"
- "보고 주기를 월 1회에서 주 1회로 변경하고 싶다"

---

## 🔍 분류 의사결정 플로우

```
신청이 들어왔다
    ↓

Q1. 명확한 오류/버그가 있는가?
    (에러 메시지, 재현 가능, 설계와 다름)
    ↓
    ✅ YES → 이슈 (즉시 개발팀 할당)
    ❌ NO → Q2로

    ↓

Q2. 고객이 새로운 기능을 원하는가?
    (새 기능 추가, 프로세스 변경, 기능 확장)
    ↓
    ✅ YES → 고객요청 (변경관리 프로세스)
    ❌ NO → Q3으로

    ↓

Q3. 어떻게 할지 결정이 필요한가?
    (불명확함, 여러 선택지, 고객과 협의 필요)
    ↓
    ✅ YES → 협의요청 (PM이 주도)
    ❌ NO → 기타 (프로젝트 관리/로깅)
```

---

## 📋 단계별 협의요청 유형

| 발생 단계 | 협의 내용 | 해결 방식 |
|-----------|----------|----------|
| 분석 | 프로세스 방식 미결정 | 고객과 선택지 협의 |
| 설계 | 기술 제약/구현 불가 | 대안 제시 후 결정 |
| 구현 | UI/UX 변경 요청 | 설계 변경 후 진행 |
| 테스트 | 미래 위험 요소 발견 | 대응 계획 수립 |
| 운영 | 요구사항 변경 요청 | 새 요청으로 처리 |

## 🎤 현업 인터뷰 분석 가이드

### 📌 왜 필요한가?

**문제 상황 예시:**
> "지금은 생산량을 수동으로 기록하는데요... 그래서 실수도 많고, 시간도 오래 걸리고... 이걸 자동으로 할 수 있으면 좋을 텐데... 근데 현재 시스템에는 이 기능이 없고..."

**문제점:**
- 여러 정보가 혼재 (현황 + 문제점 + 원하는 것 + 기술 제약)
- 어디까지가 요구사항인지 불명확
- 정확한 분류 어려움
- 의도를 놓칠 수 있음

### 1단계: 인터뷰 내용을 5가지로 분해

#### 1️⃣ 현재 운영 방식 (AS-IS)
- 누가 하는가?
- 얼마나 자주?
- 어떤 도구?

#### 2️⃣ 문제점 (Pain Points)
- 어떤 실수?
- 시간 소요?
- 비용 영향?

#### 3️⃣ 원하는 결과 (TO-BE)
- 뭘 원하는가?
- 목표 효과?
- 언제까지?

#### 4️⃣ 기술적 제약 (Technical Limits)
- 현재 시스템?
- 기계 모델?
- 호환성?

#### 5️⃣ 궁금한 점 (Questions)
- 가능한가?
- 비용은?
- 기간은?

### 2단계: 인터뷰 정리 시트 작성

```
📍 주제: 생산량 자동 수집
면담자: OOO (생산팀) | 면담일: 2026-01-18

[1] 현재 운영 방식 (AS-IS)
    담당: 생산팀 | 주기: 매시간 1회 | 도구: 엑셀 + 수동 입력

[2] 문제점
    입력 실수: 월 5~10건 (약 1%) | 소요 시간: 일일 30분

[3] 원하는 결과 (TO-BE)
    기능: 기계 → 자동 수집 → 시스템 자동 입력 | 목표: 실수율 0%, 소요시간 0분

[4] 기술적 제약
    현재 시스템: ERP (수동 입력만) | 기계: 2015년 모델 (네트워크 모듈 있음)

[5] 남은 질문
    □ 자동 연결 기술적으로 가능? □ 추가 투자 필요? □ 소요 기간?

분류: 협의요청 (기술 검토 필요)
상태: DISCUSSING | 다음액션: 기술팀 검토 → 방안 제시
```

### 3단계: 인터뷰 정리 체크리스트

**정보 수집**
- [ ] 5가지 카테고리로 분해
- [ ] 각 카테고리별 질문으로 정보 수집
- [ ] 인터뷰 정리 시트 작성
- [ ] 빠진 정보 확인

**검증 및 분류**
- [ ] 현업 검증 ("맞나요?")
- [ ] 분류 결정 (고객요청/이슈/협의요청)
- [ ] 상태 설정
- [ ] 다음 액션 명확화

### 💡 핵심 팁

**❌ 하지 말 것:**
- 즉시 분류하기
- 바로 개발 가능성 판단하기
- 부분적 정보로 결정하기
- 인터뷰 내용을 그대로 요구사항으로 변환하기

**✅ 해야 할 것:**
- 5가지 카테고리로 분해하며 듣기
- "지금 현황은요?", "구체적으로 뭔가요?" 질문하기
- 정리 시트에 기록하기
- 빠진 부분 확인하기
- 명확하게 분류하기

## ✅ 실무 처리 절차

### 고객요청 처리

**1단계: 신청 접수** (접수자)
- 요청 내용, 출처, 영향도 정리

**2단계: PM 검토** (1일)
- 계약 범위 확인
- 비용/일정 영향도 분석

**3단계: 변경관리 회의** (1주)
- 고객 + PM + 기술리드
- 범위/비용/일정 협상

**4단계: 승인**
- APPROVED → 개발 일정 추가
- REJECTED → 고객 통보

**5단계: 개발/테스트**
- IN_DEVELOPMENT → APPLIED

---

### 이슈 처리

**1단계: 이슈 등록** (발견자)
- 제목, 설명, 재현 방법, 심각도 기입

**2단계: PM 심각도 판단** (1시간)
- CRITICAL/HIGH/MEDIUM/LOW 결정
- 응답 시간 설정

**3단계: 개발팀 할당** (즉시)
- OPEN → 담당자 배정
- 원인 분석 시작

**4단계: 수정** (심각도별 기간)
- IN_PROGRESS
- 코드 수정 + 단위테스트

**5단계: QA 재검증** (1시간)
- 오류 재현 불가 확인
- RESOLVED

**6단계: 배포 및 완료**
- CLOSED

---

### 협의요청 처리

**1단계: 협의 등록** (발견자)
- DISCUSSING 상태로 등록
- 협의 주제 명확히 정의

**2단계: 분석** (기술팀/PM)
- 현황 파악
- 선택지 최소 2개 이상 도출
- 비용/일정 영향도 분석

**3단계: 고객 협의** (PM 주도)
- 각 선택지 장단점 설명
- 고객 피드백 수집

**4단계: 의사결정** (고객)
- 선택지 중 선택
- 또는 연기/거절 결정

**5단계: 변환 처리**
- 고객요청으로 변환 (REQ-XXXX)
- 또는 업무협조로 분해 (COOP-XXXX)
- 또는 일정 계획 수립
- 또는 BLOCKED 유지

---

## 🚨 자주 틀리는 분류

### ❌ 협의요청을 고객요청으로

**상황:** "지금 설계가 이렇게 되어 있는데, 고객이 다르게 하고 싶대요"

**❌ 틀린 분류:** 고객요청
- 설계 변경이 정해지지 않았는데 요청으로 등록

**✅ 올바른 분류:** 협의요청
- PM이 "다르게"의 의미 명확화
- 선택지 제시
- 고객 의사결정 후 고객요청으로 변환

---

### ❌ 협의요청을 이슈로

**상황:** "분석 결과 현재 운영 방식으로는 이 부분이 불명확합니다"

**❌ 틀린 분류:** 이슈
- 버그가 아니므로 개발팀이 해결 불가
- 책임 미루기

**✅ 올바른 분류:** 협의요청
- PM이 고객과 함께 의사결정
- 명확한 방향 결정 후 진행

---

### ❌ 이슈인데 협의요청으로

**상황:** "로그인이 안 된다" (매번 발생, 에러 메시지 있음)

**❌ 틀린 분류:** 협의요청
- 이미 버그가 명확함
- 결정할 게 없음
- 시간만 낭비

**✅ 올바른 분류:** 이슈
- 즉시 개발팀 할당
- 빠른 수정

---

### ❌ 고객요청을 이슈로

**상황:** "구형 기계라 구현이 안 되는데?"

**❌ 틀린 분류:** 이슈
- 버그가 아님
- 개발팀이 해결 불가

**✅ 올바른 분류:** 협의요청
- 기계 업그레이드 vs 대체 방안 vs 기능 포기
- 고객과 의사결정
- 고객요청 + 업무협조로 분해

---

## 📊 상태 관리 매트릭스

| 분류 | 초기 상태 | 담당자 | 다음 상태 | 완료 상태 |
|------|----------|--------|----------|----------|
| 고객요청 | REVIEWING | PM | APPROVED/REJECTED | APPLIED |
| 이슈 | OPEN | 개발팀 | IN_PROGRESS | CLOSED |
| 협의요청 | DISCUSSING | PM + 고객 | (변환) | REQ/COOP/BLOCKED |

## 💡 핵심 체크리스트

### 신청을 받았을 때

**1️⃣ 버그인가?**
- [ ] YES → 이슈로 등록 (즉시 개발팀)
- [ ] NO → 2번

**2️⃣ 새 기능인가?**
- [ ] YES → 고객요청으로 등록 (변경관리)
- [ ] NO → 3번

**3️⃣ 어떻게 할지 결정 필요한가?**
- [ ] YES → 협의요청으로 등록 (PM + 고객)
- [ ] NO → 기타

✅ **분류 완료!**

---

### 이슈 등록 전 확인

이걸 이슈로 등록하기 전에 확인:
- [ ] 재현 가능한가? (매번 같은 조건에서 발생)
- [ ] 에러 메시지/버그가 명확한가?
- [ ] 코드 수정으로 해결 가능한가?
- [ ] 고객/팀의 의사결정이 필요하지는 않은가?
- [ ] 기술팀이 해결할 수 있는 범위인가?

**모두 YES면** → 이슈로 등록
**하나라도 NO면** → 협의요청 검토

---

### 고객요청 검토 전 확인

이걸 고객요청으로 등록하기 전에 확인:
- [ ] 고객이 명시적으로 요청했는가?
- [ ] 새로운 기능/변경인가?
- [ ] 설계 변경이 필요한가?
- [ ] 비용/일정 영향이 있는가?
- [ ] 계약 범위 판단이 필요한가?

**모두 YES면** → 고객요청으로 등록
**하나라도 NO면** → 협의요청 검토

---

### 협의요청 시작 전 확인

협의를 시작하기 전에 확인:
- [ ] 버그가 아닌가?
- [ ] 명확한 기능 요청이 아닌가?
- [ ] 여러 선택지가 있는가?
- [ ] 고객/팀 의사결정이 필요한가?
- [ ] 기술적/비즈니스적 영향도가 큰가?

**모두 YES면** → 협의요청으로 등록
- 선택지 3개 이상 도출
- PM이 고객과 협의
- 최종 결정 후 변환

---

## 🎯 실무 예시

### 예시 1: 구형 설비의 기능 제약

**상황:** "고객이 실시간 데이터 수집을 원하는데 구형 기계(2015)에는 네트워크 모듈이 없습니다"

**분류:** ✅ 협의요청 (기술 제약)

**처리:**

**1) PM이 기술팀과 분석**
- 기계 명세 확인
- 업그레이드 비용/기간
- 대체 방안 검토

**2) PM이 고객과 협의**
- 옵션 A: 기계 업그레이드 (₩20M, 4주)
- 옵션 B: 수동 입력 (비용 0)
- 옵션 C: 기능 포기

**3) 고객 선택**
→ 옵션 A 선택

**4) 변환 처리**
- 고객요청 생성: REQ-2026-001 (실시간 데이터 수집 기능)
- 업무협조 생성: COOP-2026-001 (기계 네트워크 모듈 설치)

**결과:** 명확한 실행 계획 수립

## 🚀 성공 팁

### 1. 분류 시 가장 중요한 3가지

**① 버그인가?** (재현 가능, 명확한 오류)
→ YES → 이슈

**② 새로운 기능인가?** (고객이 원함, 설계 변경)
→ YES → 고객요청

**③ 결정이 필요한가?** (여러 선택지, 의사결정 필요)
→ YES → 협의요청

---

### 2. 협의요청이 나왔을 때

**절대 하면 안 되는 것:**
- ❌ 그냥 진행
- ❌ 개발팀에 일임
- ❌ 고객 확인 없이

**꼭 해야 하는 것:**
- ✅ PM이 주도
- ✅ 선택지 제시
- ✅ 고객과 협의
- ✅ 명확한 결정
- ✅ 다음 액션 명확화

---

### 3. 상태 관리

- **이슈:** OPEN → CLOSED 빨리
- **고객요청:** REVIEWING → 명확한 결정
- **협의요청:** DISCUSSING → 다른 상태로 변환

**⚠️ 대기 상태가 생기면 이유 명시!**

## 📞 Q&A

**Q: 분류가 애매한데?**
A: PM에게 물어보세요. 초기 분류가 틀려도 괜찮습니다. 상태를 변경할 수 있으니까요.

**Q: 협의가 길어지면?**
A: 협의 기간을 정해두세요 (3~5일). 기한까지 결정 못 하면 PM이 주도적으로 의사결정.

**Q: 변환된 항목 추적은?**
A: 협의요청 → 고객요청 변환 시 원래 협의요청 번호를 고객요청에 참고로 기록.

**Q: 이슈가 많으면?**
A: 심각도별로 처리합니다. CRITICAL은 즉시, MEDIUM은 1주일.
